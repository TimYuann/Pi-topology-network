## 总体结论

我认可这份 brief 的阶段判断：**T1–T5 已经足够 coherent，可以进入下一道“pre-effectful”工程 gate；但仍不应进入真实 spawn、signal、delete、cleanup 或 v0.5 runtime integration。**

T5 已经把上一轮要求的 before-effects hardening 基本补齐：Foundation-0 canonical storage、`ResourceCreationPlan`、verified never-created abandoned semantics、parent-directory fsync、rich lock metadata、conservative stale-lock recovery 都已实现；同时明确没有做真实 spawn、signal、delete、managed temp-directory creation、Ghostty/Pi topology spawn 或 v0.5 integration。

当前唯一硬阻塞判断也成立：**Durable cleanup-attempt acquisition 必须在任何真实 signal/delete/temp cleanup 前完成**；brief 已明确 T3 的 in-memory coordination 只适用于 pure lifecycle / read-only phases，不足以支持真实 cleanup execution。

我建议下一阶段正式定为：

```text
T6: Durable Cleanup-Attempt Acquisition
Scope: no real cleanup effects
```

也就是说，T6 仍然应该是控制面能力，不是 effectful runtime 行为。

# 1. T1–T5 pre-effects foundation 是否足够 coherent？

**结论：是，可以通过。**

当前实现顺序仍然正确：

```text
schema contract
-> mission event lock and canonical append
-> managed resource lifecycle and pre-registration
-> read-only process inspection
-> before-effects hardening
```

brief 明确说明 Foundation-0 仍然隔离在 `packages/pi-topology/src/runtime/foundation0/`，没有接入 v0.5 runtime，也没有发起真实进程、信号、目录删除或 cleanup。

这说明目前工程没有越过设计边界。尤其 T5 补齐了几项之前阻塞 effectful work 的关键问题：

```text
canonical foundation0 storage
ResourceCreationPlan
verified never-created abandoned
parent-directory fsync
rich lock metadata
conservative stale-lock recovery
```

这些正是进入真实 resource creation / cleanup 前必须完成的硬化项。

当前 verification 也足够支持进入下一道非 effectful gate：Foundation-0 单元、集成、typecheck 均通过，且 forbidden signal/spawn scan 通过。

但这只能说明：

```text
pre-effects foundation coherent
```

不能说明：

```text
ready for real cleanup execution
```

后者仍被 durable cleanup-attempt acquisition 阻塞。

# 2. T6 是否应严格是 Durable Cleanup-Attempt Acquisition？

**结论：是。T6 应严格做 durable cleanup-attempt acquisition，并且不得包含真实 spawn、signal、delete 或 cleanup。**

我建议接受 HQ 的推荐：下一项实现任务就是：

```text
T6: Durable Cleanup-Attempt Acquisition
```

并保持它是 narrow reviewed capability。

原因是 first-slice contract 已经要求：

```text
At most one cleanup attempt may be active for the same resource_id + identity_digest.
Cleanup-attempt acquisition MUST be serialized.
Concurrent cleanup with a different idempotency key MUST return cleanup_in_progress.
```



在这件事没有 durable 化之前，任何真实 signal/delete 都缺少最关键的并发防线。

我建议 T6 的合同范围写成：

```text
T6 MUST:
- acquire cleanup attempt under mission event lock or equivalent durable serialization
- key by mission_id + resource_id + identity_digest
- support same idempotency key replay
- reject different active idempotency key as cleanup_in_progress
- persist ActionRequest / ActionAttempt / PolicyDecision / acquisition outcome
- rebuild active cleanup-attempt state from canonical events
- route interrupted or indeterminate attempts to reconciliation

T6 MUST NOT:
- send SIGTERM
- send SIGKILL
- delete temp directories
- rename temp directories to quarantine
- spawn managed processes
- create managed temp directories
- integrate into v0.5 live runtime
```

This keeps T6 aligned with the current brief: acquisition, policy decision, identity check, and final or indeterminate outcome must be replayable from canonical storage, but no real signal/delete/temp cleanup is included unless separately authorized.

# 3. Temp-directory creation/cleanup 是否应排在 process cleanup 前？

**结论：是，建议先做 temp-directory path。**

推荐顺序：

```text
T6  Durable cleanup-attempt acquisition
T7  Temp-directory creation identity + marker
T8  Temp-directory quarantine/delete guard
T9  Process creation identity
T10 Process signal guard
T11 Reconciliation hardening
T12 Closeout integration
```

理由是 temp-directory cleanup 虽然也危险，但它的边界比 process signaling 更容易做成封闭系统：

```text
approved temp root
marker file
identity digest
realpath/lstat
quarantine rename
no symlink following
protected path rejection
```

first-slice contract 已经把 temp-directory cleanup 的安全条件写得比较具体，包括 approved root、realpath、marker 校验、禁止 symlink、禁止 root/runtime/mission/repo/current cwd 等路径，并要求 quarantine 后再删除。

Process cleanup 的风险更高，因为 signal 一旦发错，可能影响用户当前 CLI、ancestor、同 process group 或 PID reuse 后的无关进程。contract 明确 PID alone 不够，PID 存在但 identity mismatch 必须 skip 且不得 signal，当前 CLI、祖先进程和包含 CLI 的 process group 都不得 signal。

所以建议：

```text
Temp directory: creation -> marker -> quarantine/delete
Process: creation identity -> exact identity proof -> signal guard
```

不要先做 process signal。

# 4. macOS ProcessInspector 的限制是否意味着 process cleanup 前需要更强 identity source？

**结论：是。当前 T4 只能支持 read-only fact collection，不能支持 signal authorization。**

brief 已明确：host macOS probe 在 argv 或 start-time precision ambiguous 时会降级为 non-exact result；T4 不授权 cleanup，只提供 read-only process facts。

因此 process cleanup gate 必须采用：

```text
non-exact identity = no signal
```

不能把 partial identity 当成“足够接近”。

Process cleanup 前至少需要补齐：

```text
raw OS start tuple with sufficient precision
pid + pgid
current CLI pid / ancestors / process group protection
executable realpath
cwd realpath if used in command_digest
argv observation and truncation behavior
command_digest canonicalization
permission-denied behavior
process-exited-during-probe behavior
zombie/unsupported behavior
```

并且必须明确：

```text
permission_denied
partial_identity
ambiguous_start_time
argv_unavailable
process_exited_during_probe
unsupported_platform
```

这些都不是 allowed-to-signal 状态。

此外，contract 当前只要求 force kill 前重查 identity 和 CLI protection。 我建议后续 process-signal gate 强化为：

```text
Identity, authorization, mission phase, cleanup policy, and CLI protection
MUST be rechecked immediately before every signal step.
```

包括：

```text
SIGTERM
SIGKILL
process-group SIGTERM
process-group SIGKILL
```

# 5. T5 stale-lock recovery 是否足够？timing flake 是否要在 T6 前稳定？

**结论：T5 的保守策略足够；但 timing flake 必须在 T6 之前稳定或隔离。**

策略本身我认可。T5 stale-lock recovery 已经足够保守：

```text
malformed metadata: do not break
invalid created_at: do not break
hostname mismatch: do not break
missing holder probe: do not break
permission denied / ambiguous / unsupported / present matching: do not break
verified absent or verified start-tuple mismatch: can break only through explicit stale recovery
verified stale break reports an incident before removal
```



这符合 effectful path 的安全原则：

```text
cannot verify stale => fail safe
```

不过，brief 记录了 full foundation0 test run 出现一次 stale-lock timing flake，随后 immediate rerun 通过。

我不认为这个 flake 阻塞“review approval”，但它应该阻塞 T6 merge，除非满足以下之一：

```text
A. flake root cause 已定位并修复；
B. test 被重写为 deterministic fake clock / fake holder probe；
C. flake 被隔离为非语义性环境时序问题，并有重复 stress run 证明不影响 lock correctness。
```

建议 T6 前加一条小 gate：

```text
stale-lock tests must be deterministic under repeated local runs
```

尤其 T6 将把 lock/acquisition 作为 cleanup 的核心串行化机制，不能带着 timing flake 进入 acquisition layer。

# 6. 后续 effectful work 是否应拆成多个小 gate？

**结论：必须拆。不要把 creation、cleanup、reconciliation、closeout 合成一个大 T6/T7。**

建议 gate 切分如下。

## T6: Durable Cleanup-Attempt Acquisition

```text
No real effects.
Only durable claim / conflict / idempotency / replay.
```

验收重点：

```text
same key -> existing state
different key while active -> cleanup_in_progress
crash before outcome -> reconstruct active/interrupted attempt
indeterminate -> reconciliation_required
```

## T7: Temp-Directory Creation Identity + Marker

```text
Allowed effect: create managed temp directory under approved temp root.
No delete.
```

验收重点：

```text
ResourceCreationPlan durable before creation
marker written and verified
identity_digest stable
planned -> registered -> active
crash after creation before activation reconciles
```

## T8: Temp-Directory Quarantine/Delete Guard

```text
Allowed effect: quarantine and delete only registered owned temp directory.
No process signal.
```

验收重点：

```text
marker mismatch -> no delete
symlink escape -> no delete
replacement race -> no delete
quarantine path durable before rename
crash after rename -> recover locator
```

## T9: Process Creation Identity

```text
Allowed effect: spawn runtime-owned process.
No signal.
```

验收重点：

```text
planned resource before spawn
observed pid/pgid/start tuple
dedicated process group when requested
CLI/self protection facts recorded
non-exact identity blocks future cleanup
```

## T10: Process Signal Guard

```text
Allowed effect: signal only exact registered process identity.
```

验收重点：

```text
no broad matching
PID reuse skip
ancestor/CLI/process-group skip
SIGTERM recheck
SIGKILL recheck
authorization revoked after acceptance blocks execution
```

## T11: Reconciliation

```text
Resolve interrupted / indeterminate cleanup without blind retry.
```

验收重点：

```text
after effect before outcome
after quarantine before delete
after process absent but no outcome
missing payload/digest mismatch
```

## T12: Closeout Integration

```text
Clean closeout only when resources are cleaned/verified or never-created abandoned.
```

验收重点：

```text
closing rejects new register_resource
snapshot digest binds owner decision
conditional closeout records residual risk
clean closeout blocked by unverified residuals
```

This matches the first-slice acceptance-test spirit: revoked authorization before execution must deny cleanup, concurrent cleanup must allow only one active attempt, crash after effect must reconcile, and no external effect may occur before intent and execution-boundary allowed decision are durably committed.

# 7. 是否还有 ResourceCreationPlan 的 path / digest / fsync / replay gap？

**结论：T5 方向正确，但 resource creation 前还需要再确认 6 个细节。**

T5 已经引入 `ResourceCreationPlan`，记录 planned resource、cleanup policy、creation kind、creation payload、authorization、requesting action、timestamp 和 canonical effect fingerprint；runtime helper 会重算 effect fingerprint，不信任调用方 digest hint。

这已经关闭了上一轮最大的 sidecar durability gap。
但在允许 real creation 前，我建议逐项确认：

## 7.1 `ResourceCreationPlan` 是否进入 canonical event replay

要求：

```text
resource_planned event payload must contain or reference the complete ResourceCreationPlan.
Projection rebuild must be able to reconstruct:
- planned resource
- cleanup policy
- creation kind
- creation payload
- authorization
- effect_fingerprint
```

仅能验证 payload digest 不够；replay 必须能恢复创建意图。

## 7.2 `ResourceCreationPlan` 的 fingerprint 输入必须稳定

建议明确 fingerprint 输入为：

```text
schema_version
mission_id
resource_id
resource_type
planned_resource canonical JSON
cleanup_policy canonical JSON
creation_kind
creation_payload canonical JSON
authorization_id
requesting_action_id
```

不要包含会在重放时变化的派生字段，例如 wall-clock observation、event_id、sequence、payload path。

## 7.3 `payload_ref` 应保持相对 Foundation-0 storage

brief 当前规定 event payload refs 使用：

```text
foundation0/payloads/<payload_digest>.json
```



我认可这个选择。它避免把 mission root 和 foundation0 subdir 混用。后续所有 event/projection/recovery 都应只通过：

```text
foundation0StoragePaths(missionDir)
```

构造路径，不要手写字符串。

## 7.4 Durable ordering 需要覆盖 “first creation of directories”

T5 已加入：

```text
payload temp file fsync
rename
payload directory fsync
runtime event file fsync
foundation0 directory fsync when event log first created
lock metadata fsync
locks directory fsync
projection atomic JSON write
```



建议确认还包括：

```text
foundation0 directory newly created -> parent missionDir fsync
payloads directory newly created -> foundation0 dir fsync
locks directory newly created -> foundation0 dir fsync
```

因为第一次创建目录本身也有 directory-entry durability 问题。

## 7.5 Missing payload / digest mismatch 必须进入 reconciliation

如果 event 存在但 payload 缺失或 digest mismatch，不能生成成功 projection。

应返回：

```text
reconciliation_required
```

或：

```text
unsupported_schema / corrupt_payload incident
```

并阻止 resource creation / cleanup execution。

## 7.6 CreationPlan 与 Authorization drift

`ResourceCreationPlan` 记录了 authorization，但真实 creation 仍必须在 execution boundary 重新验证 authorization。contract 已要求 action intent 与 execution-boundary allowed `PolicyDecision` 必须在外部 effect 前 durable commit。

所以 creation gate 不能只检查：

```text
plan has authorization_id
```

而必须检查：

```text
current authorization chain valid
mission phase active
policy hash not stale
execution PolicyDecision allowed
```

# 对 HQ recommendation 的最终判断

我同意 HQ 的建议：

```text
Do not proceed directly to real spawn, signal, delete, or v0.5 runtime integration.
```

并同意下一步是：

```text
T6: Durable Cleanup-Attempt Acquisition
```

T6 完成前，不应允许真实 cleanup。
我进一步建议：**T6 完成前也不要做真实 resource creation**，因为 creation 一旦成功，就可能立刻产生需要 cleanup 的 residual resource；而 cleanup-attempt acquisition 仍是当前唯一硬阻塞。brief 本身也建议第一个 effectful Foundation-0 task 应继续被 durable acquisition 阻塞。

最终 gate 判断：

```text
T1–T5: approve
T6 durable cleanup-attempt acquisition: approve as next task
real temp-directory creation: blocked until T6 passes
real temp-directory delete: blocked until T6 + temp identity/marker gate
real process creation: blocked until T6 + process creation identity gate
real process signal: blocked until stronger exact identity + signal guard gate
v0.5 runtime integration: blocked until at least temp/process effectful gates have passed in isolation
```

最重要的一句话：

```text
可以继续向 effectful Foundation-0 work 推进，
但下一步仍必须是无副作用的 durable acquisition gate，
不是直接进入真实创建或清理。
```
