# Pi拓扑网络：三方技术探讨导读

Date: 2026-06-26
Status: discussion intro
Audience: owner / Codex / Pi

## 1. 项目一句话

Pi拓扑网络是一个面向多 Agent 协作实践的运行协议项目。它试图解决的问题不是“如何让一个 Agent 更聪明”，而是“如何让多个能力不同、职责不同、生命周期不同的 Agent 可控地一起工作”。

它的核心判断是：生产级 agentic coding 不应只依赖单一对话窗口，也不应把所有 Agent 放进无约束的自由聊天网络。更合理的形态是：

```text
mesh communication + centralized authority + single-writer implementation + independent review
```

也就是：信息可以横向流动，权力必须纵向收口；执行面尽量保持单写者，审查面保持独立判断。

## 2. 这个项目在抽象上想沉淀什么

项目沉淀的不是某个具体任务的过程记录，而是一套可复用的协作秩序：

- 如何把 owner 的目标转成可执行的 mission。
- 如何把角色拆成 supervisor、hq、oracle、repair、runner 等稳定职责。
- 如何区分“我收到了”和“我完成了”。
- 如何让报告、证据、审查、修复、验证各自有清楚的归属。
- 如何在多 Agent 并行时避免权限扩散、上下文污染、消息阻塞和重复劳动。

因此，Pi拓扑网络更像一个“多 Agent 项目操作系统”的雏形：它关心角色、边界、状态、证据、路由和生命周期，而不是只关心单次 prompt 的质量。

## 3. 当前的基本协作模型

现有模型把 Agent 分成两类：

| 类型 | 生命周期 | 主要价值 |
|---|---|---|
| Persistent role mesh | 长驻 | 保持连续上下文、角色责任和审计链路 |
| Ephemeral task swarm | 短生命周期 | 承担局部搜索、验证、复现、一次性分析 |

长驻角色负责“秩序”：谁有权决策，谁负责合并，谁负责审查，谁负责修复，谁负责验证。短生命周期任务负责“吞吐”：快速完成局部工作，但不直接改变系统级权力结构。

目前默认角色大致是：

- `topology-supervisor`：owner-facing 的入口与运行控制面。
- `hq`：开发编排者，负责拆解、派发、合并和阶段判断。
- `oracle`：独立审查者，负责风险、证据质量和放行判断。
- `repair`：受限执行者，只在明确授权范围内修改。
- `runner`：验证者，负责测试、smoke、复现和证据记录。
- `governor`：历史 owner-facing 入口，在部分旧流程中保留。

这套模型强调：横向通信传递信息，不传递授权。权限、范围、止损和最终 verdict 必须回到 supervisor / governor / hq 这类权威节点。

## 4. 当前实现方式的高层轮廓

当前实现已经从纯文档协议推进到本地 Pi runtime / extension 形态。它大致由几层组成：

1. 协议层：定义 ACK、REPORT、STATUS、MISSION UPDATE、AUTHORIZATION 等消息类型，以及角色边界和状态机。
2. 任务层：以 Mission 为单位保存目标、状态、运行事件、证据和 closeout。
3. 角色层：为不同 Agent 提供角色 prompt、启动脚本和工作边界。
4. 传输层：使用本地文件 / JSONL 作为已验证的消息与事件记录方式。
5. 控制层：通过 Pi extension / tool 调用把初始化、派生、发送、等待、读取、写 artifact 等动作纳入统一入口。
6. 终端层：目前使用 Ghostty 启动多个角色窗口或 session，让这些角色作为可观察的独立运行实例存在。

从概念上看，当前实现不是一个“中心调度器控制一切”的系统，而是一个“有中心权威的角色网络”。中心节点负责授权和合并，边缘角色负责专业动作，消息和证据通过运行时协议被记录下来。

## 5. 现有方式已经证明的价值

这个项目的价值来自实战压力，而不是先验设计。

已有实践表明，多 Agent 协作中最容易出问题的地方不是“某个模型不会写代码”，而是：

- 收到任务后没有及时 ACK，导致上游误判卡住。
- 把业务报告塞进原始回复通道，导致消息通道语义混乱。
- 等待窗口过长，把异步协作变成阻塞等待。
- 审查者和修复者角色混用，导致独立性消失。
- 测试者同时修改代码，导致证据不再干净。
- 横向角色互相传递“隐式授权”，导致 scope 漂移。
- 长输出直接灌回主会话，污染上下文并放大延迟。

Pi拓扑网络目前通过 direct ACK、artifact-first、角色边界、Mission 状态机和 owner gate，把这些问题显式化，并把它们变成可以审查和改进的运行规则。

## 6. 当前正在暴露的新问题

随着项目从协议验证进入实际运行，问题开始从“Agent 是否知道该怎么协作”转向“runtime 是否能可靠约束协作”。

当前最值得讨论的是两类问题。

第一类是实操层面的编排问题：软约束和硬约束如何配合。

软约束包括 prompt、角色说明、计划文档、任务包、协作规范和 reviewer 纪律。它们能塑造意图，但不能完全防止越权、误写、误杀进程或错误路由。

硬约束包括 tool guard、路径权限、角色写入策略、launch metadata、process ledger、消息 schema 校验、artifact API 和 cleanup owner。它们能把协作纪律从“建议”变成“运行时边界”。

第二类是抽象层面的通信问题：如何避免多 Agent 协作中的通道阻塞和延迟。

当前方向是把 lifecycle ACK 和 business report 分开：ACK 快速完成原请求，真正的业务结果通过结构化 packet 和 artifact 异步返回。这样可以避免一个角色长时间占住上游等待窗口，也避免把长报告直接塞进当前会话。

## 7. 关于 Ghostty 与进程生命周期

当前 Ghostty 多窗口启动方式让每个角色以可见独立实例运行，便于观察和调试。但它也带来两个现实问题：

- 使用强制新应用实例的启动方式时，macOS Dock 会出现多个 Ghostty 图标。
- 测试或 dogfood 完成后，部分角色进程不会自动归属和清理，需要人工跟进 Ctrl+C。

这说明“启动方式”只是表层问题，更深层的问题是“运行实例是否有明确 owner”。一个健康的拓扑 runtime 应该知道每个角色进程属于哪个 mission、由哪个动作启动、对应哪个脚本、应该由谁关闭、什么情况下可以安全终止。

因此，后续讨论可以先把重点放在进程所有权和清理边界上，而不急着把所有启动方式一次性替换掉。

## 8. 三方探讨建议聚焦

这次三方技术探讨建议围绕四个问题展开：

1. 软编排应该注入到哪里？
   例如角色 prompt、mission card、plan doc、inline task、packet header、artifact 模板分别承担什么职责。

2. 硬编排应该卡在哪里？
   例如 tool_call interception、路径写入策略、role capability、process ownership、cleanup guard、message schema、owner gate 分别由哪一层执行。

3. 消息如何既不丢失又不阻塞？
   核心是 ACK 与业务报告分离、artifact-first、reply_target、correlation_id、deadline_hint、stale / dedupe / backpressure 机制。

4. 多 Agent 协作的“权力流”和“信息流”如何分离？
   信息可以 mesh，授权必须收口。系统要允许角色之间高效交换证据，但不能允许证据交换自动变成 scope 扩大或最终 verdict。

## 9. 一个可供讨论的目标状态

理想状态下，Pi拓扑网络应该成为一套可以反复用于真实项目的协作底座：

- owner 只需要描述目标，不需要手动追每个 Agent。
- supervisor 能把目标转成 mission，并维护全局状态。
- hq 能拆解和合并，但不越过 owner gate。
- oracle 能独立审查，不被执行链污染。
- repair 能安全修改，不越权扩散。
- runner 能提供干净证据，不承担代码责任。
- 所有长报告和测试输出优先落 artifact，主通道只传摘要和引用。
- 所有进程、消息、事件和证据都有 mission 归属。
- 测试结束后，系统能清理自己启动的东西，同时不误伤用户正在使用的 CLI 本体。

这不是为了制造一个更复杂的 Agent 系统，而是为了把复杂度放在应该被管理的位置：角色边界、运行状态、证据链、权限收口和生命周期管理。
