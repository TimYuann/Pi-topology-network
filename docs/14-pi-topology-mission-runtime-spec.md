# Pi Topology Mission Runtime Spec

Date: 2026-06-17
Project: OMP拓扑网络 / `packages/pi-topology`
Status: reviewer draft; M3 official-doc review incorporated
Depends on: `docs/13-pi-topology-mission-runtime-prd.md`

## 1. Scope

This spec translates the Mission Runtime PRD into implementation contracts for `packages/pi-topology`.

The implementation target is a small, reviewable extension of the current local runtime:

- keep Pi visible peer sessions
- keep local JSONL/file transport as the verified runtime
- add Mission registry and per-Mission storage
- preserve root `.pi/topology/*` compatibility for the active Mission during migration
- distinguish launch request, resumability, liveness, staleness, parking, and closure
- keep direct generated-script launch as the stable dogfood lane

Official Pi API alignment is intentionally bounded here. This spec labels primitives using:

- `supported`: already used or verified in local records
- `pending_api_audit`: must be checked in the next API audit before implementation relies on it
- `compatibility_target`: desired future alignment, not a current contract
- `local_protocol`: OMP拓扑网络 file/JSONL protocol layered on top of Pi
- `local_environment_risk`: behavior depends on this Mac or terminal layer, not on Pi core APIs

## 2. Current Baseline

The current runtime writes active Mission files directly under `.pi/topology/`:

```text
.pi/topology/
  mission-card.json
  status-board.json
  runtime-events.jsonl
  incident-log.jsonl
  sessions.jsonl
  launch/
    topology-supervisor.sh
    hq.sh
    repair.sh
    runner.sh
    oracle.sh
    librarian.sh
    scott.sh
  artifacts/
```

The current verified packet transport writes outside the project state directory by default:

```text
<PI_COMS_DIR or /tmp/pi-topology-<project>>/
  projects/<project>/agents/<role>.json
  projects/<project>/packets/outbox.jsonl
  projects/<project>/packets/<role>-inbox.jsonl
```

The redesign must adapt this baseline. It must not require a one-shot destructive migration.

Raw packet transport remains shared by project and role inbox. Per-Mission `packet-ledger.jsonl` is a derived Mission-local index written when packets are sent, read, classified, or cleaned up. The first implementation should use read-time filtering by `mission_id` plus derived Mission ledgers; it does not reroute raw transport files into Mission folders.

`<slice_id>-notes.md` is the canonical slice handoff file. It must include changed files, focused test commands, evidence paths, commit hash, and known risks. `<slice_id>-smoke.log` stores raw smoke output when available.

### 3.2 Compatibility Files

The root files remain compatibility mirrors for the active Mission during the migration period:

- `.pi/topology/mission-card.json`
- `.pi/topology/status-board.json`
- `.pi/topology/runtime-events.jsonl`
- `.pi/topology/incident-log.jsonl`
- `.pi/topology/sessions.jsonl`
- `.pi/topology/launch/*`
- `.pi/topology/artifacts/*`

The canonical per-Mission files live under `.pi/topology/missions/<mission_id>/`.

During migration, root files may be canonical only when no `mission-registry.json` exists. Once the registry exists, root files must mirror the active Mission and must not be used as the source of truth for archived or inactive Missions.

### 3.3 Active Mission Pointer

`.pi/topology/active-mission.json` must contain:

```json
{
  "version": 1,
  "mission_id": "project-2026-06-17-001",
  "mission_dir": ".pi/topology/missions/project-2026-06-17-001",
  "selected_at": "2026-06-17T00:00:00.000Z",
  "selected_by": "topology-supervisor",
  "reason": "created|resumed|owner_selected|migration",
  "event_id": "evt_..."
}
```

Changing the active pointer must append a `mission_selected` runtime event to the selected Mission and a `registry_updated` event to the registry audit stream if such a stream is added later.

`event_id` must use the runtime event id generated by the event ledger. Current `evt_<uuid>` ids are acceptable; implementations may later move to time-sortable ids if migration preserves existing references.

### 3.4 Mission Registry

`.pi/topology/mission-registry.json` must contain:

```json
{
  "version": 1,
  "active_mission_id": "project-2026-06-17-001",
  "updated_at": "2026-06-17T00:00:00.000Z",
  "missions": [
    {
      "mission_id": "project-2026-06-17-001",
      "mission_dir": ".pi/topology/missions/project-2026-06-17-001",
      "title": "short human title",
      "objective": "full objective",
      "lifecycle_state": "awaiting_owner_confirmation",
      "progress_status": "awaiting_owner_confirmation",
      "owner_gate": "required|clear",
      "blocked": false,
      "archived": false,
      "last_updated_at": "2026-06-17T00:00:00.000Z",
      "role_summary": {
        "live": 1,
        "resumable": 0,
        "stale": 0,
        "parked": 0,
        "closed": 0
      },
      "pending_packet_count": 0,
      "incident_count": 0,
      "closeout_path": null
    }
  ]
}
```

Registry summaries are derived views. The per-Mission files remain the detailed evidence.

`lifecycle_state` is the Mission-level state machine value used by the dashboard. `progress_status` mirrors the current mission-card progress value for backward compatibility and migration visibility; it must not override `lifecycle_state`.

## 4. State Machines

### 4.1 Mission

Target Mission lifecycle states:

- `draft`: Mission card exists, owner has not approved.
- `awaiting_owner_confirmation`: Supervisor is ready and owner gate is active.
- `team_building`: owner approved the Mission and initial role launch/resume is in progress.
- `running`: role work is active.
- `reviewing`: implementation or evidence is complete enough for review.
- `delivering`: Supervisor/HQ is preparing owner-facing closeout.
- `delivered`: closeout exists and owner ack event is recorded.
- `archived`: Mission is closed for normal dashboard purposes but remains inspectable.
- `blocked`: progress is stopped pending owner/reviewer decision.
- `rollback_pending`: a slice or Mission action failed and rollback requires explicit decision.
- `parked`: Mission is intentionally paused.
- `abandoned`: owner explicitly stops the Mission without delivery.

Current `MissionProgress.status` values map as follows:

| Current status | Target lifecycle |
| --- | --- |
| `draft` | `draft` |
| `awaiting_owner_confirmation` | `awaiting_owner_confirmation` |
| `supervisor_ready` | `awaiting_owner_confirmation` |
| `running` | `team_building` or `running`, depending on peer state |
| `blocked` | `blocked` |
| `completed` | `delivered` |
| `abandoned` | `abandoned` |

Every lifecycle transition must append a `mission_lifecycle_transition` event with:

- `mission_id`
- `from_state`
- `to_state`
- `reason`
- `actor`
- `owner_decision_id`, when owner action caused the transition
- `evidence.transport`
- `evidence.business`
- `evidence.inference`

### 4.2 Session

Session states:

- `planned`: role is part of Mission policy but no script exists.
- `script_written`: launch script exists; not evidence of a live role.
- `launch_printed`: direct command was prepared; not evidence of a live role.
- `launch_requested`: terminal open command was issued; not evidence of a live role.
- `alive_confirmed`: role self-confirmed with session id, registry heartbeat, runtime event, or packet evidence.
- `live`: derived dashboard state from fresh liveness evidence.
- `resumable`: known session id or script exists, no closed/failed evidence, but fresh liveness is missing.
- `stale`: historical evidence exists but freshness window expired.
- `parked`: owner/runtime intentionally paused this role.
- `closed`: role closed normally.
- `failed`: launch or role failed.

Dashboard and registry rollups must still use the PRD five-state classification: `live`, `resumable`, `stale`, `parked`, `closed`. Internal evidence states such as `script_written`, `launch_printed`, and `launch_requested` must roll up to `resumable` or `stale` with an explanatory launch step, not appear as extra top-level dashboard categories.

Liveness rules:

- A registry heartbeat is `live` only while `readFreshPeerRegistrySync` or equivalent marks it fresh. Current code uses a 20 second default for registry freshness.
- A session record may be `resumable` for up to the Mission resume freshness window. Default: 10 minutes after the latest usable session event.
- A session record older than the resume freshness window is `stale` unless owner explicitly parks it.
- `script_written`, `launch_printed`, and `launch_requested` are never `live`.
- A closed or failed record overrides older resumable records for the same role/session id.

### 4.3 Task

Task records are introduced as Mission-local state, not as a replacement for packets.

Task states:

- `proposed`
- `approved`
- `dispatched`
- `acknowledged`
- `in_progress`
- `reported`
- `reviewing`
- `accepted`
- `needs_repair`
- `blocked`
- `closed`

Initial implementation may keep task state inside status board `active_slice` and packet lifecycle records. If a task ledger is added, it should live at `.pi/topology/missions/<mission_id>/task-ledger.jsonl`.

### 4.4 Packet

Packet states:

- `queued`
- `delivered`
- `acknowledged`
- `in_progress`
- `reported`
- `report_acknowledged`
- `closed`
- `ignored`
- `stale`
- `duplicate`
- `preserved`

`ignored`, `stale`, and `duplicate` are active-read classifications. They must not delete the packet from the raw transport outbox or inbox.

Packet records in `packet-ledger.jsonl` must include:

- `packet_id`
- `mission_id`
- `type`
- `from`
- `to`
- `request_msg_id`
- `correlation_id`
- `state`
- `raw_transport_path`
- `first_seen_at`
- `last_seen_at`
- `classification_reason`
- `artifact_path`, when the packet references an artifact

### 4.5 Artifact

Artifact states:

- `draft`
- `written`
- `referenced`
- `reviewed`
- `accepted`
- `superseded`
- `archived`

Artifacts must remain under the Mission folder. Packets should reference artifacts by mission-scoped relative path.

### 4.6 Incident

Incident states:

- `open`
- `acknowledged`
- `mitigating`
- `resolved`
- `escalated`
- `closed`

Incident records must separate:

- transport evidence
- business evidence
- inference

### 4.7 Transition Rules

Every state change must append a transition event before or with the state write. Missing transition events are reviewer findings.

Minimum transition events:

| Machine | Event | From | To |
| --- | --- | --- | --- |
| Mission | `mission_initialized` | none | `draft` or `awaiting_owner_confirmation` |
| Mission | `owner_approved_mission` | `awaiting_owner_confirmation` | `team_building` |
| Mission | `role_alive_confirmed` | `team_building` | `running` |
| Mission | `review_requested` | `running` | `reviewing` |
| Mission | `delivery_started` | `reviewing` | `delivering` |
| Mission | `owner_acknowledged_delivery` | `delivering` | `delivered` |
| Mission | `mission_archived` | `delivered` or `abandoned` | `archived` |
| Mission | `rollback_requested` | any non-archived state | `rollback_pending` |
| Mission | `mission_parked` | any non-archived state | `parked` |
| Session | `launch_scripts_written` | `planned` | `script_written` |
| Session | `spawn_result(mode=print)` | `script_written` | `launch_printed` |
| Session | `spawn_result(mode=launch)` | `script_written` or `launch_printed` | `launch_requested` |
| Session | `session_alive` | `launch_requested`, `launch_printed`, `resumable`, or `stale` | `alive_confirmed` |
| Session | `fresh_registry_seen` | `alive_confirmed` | `live` |
| Session | `freshness_expired` | `alive_confirmed`, `live`, or `resumable` | `stale` |
| Session | `role_parked` | `live`, `resumable`, or `stale` | `parked` |
| Session | `role_closed` | any non-failed state | `closed` |
| Packet | `packet_sent` | none | `queued` or `delivered` |
| Packet | `ack_received` | `queued` or `delivered` | `acknowledged` |
| Packet | `status_received` | `acknowledged` | `in_progress` |
| Packet | `report_received` | `in_progress` or `acknowledged` | `reported` |
| Packet | `report_ack_received` | `reported` | `report_acknowledged` and `closed` |
| Packet | `packet_classified_stale` | any non-closed state | `stale` |
| Packet | `duplicate_seen` | any state | `duplicate` |

Artifact and Incident state transitions may be implemented after Mission, Session, and Packet transitions, but every artifact or incident state write must still carry `event_type`, actor, and evidence.

## 5. Supervisor Mission Choice

### 5.1 Bare `/topology`

When owner runs `/topology`, Supervisor must load registry state in this order:

1. If `mission-registry.json` exists, load it.
2. Else if root `mission-card.json` exists, treat it as a legacy active Mission and offer migration/resume.
3. Else inspect latest assistant task card or show intake.

The picker/dashboard is a read-only snapshot at fetch time. Mutations to other Missions while the picker is displayed do not roll back the owner's choice. Any later command must re-read state before mutating.

### 5.2 Mission Categories

The dashboard must distinguish:

- `new`: no Mission exists or owner supplied a new objective.
- `active`: matches `active-mission.json` and is not archived.
- `resumed`: owner selected an existing non-archived Mission.
- `archived`: closed for normal work, inspectable only.
- `blocked`: requires owner/reviewer decision before work continues.
- `parked`: intentionally paused.

### 5.3 Owner Actions

Supported owner actions:

- continue active Mission
- resume a selected Mission
- create new Mission
- inspect Mission
- archive Mission
- park Mission
- unpark Mission
- mark blocked
- request rollback

Actions that mutate state must append runtime events and update `mission-registry.json`.

## 6. Role Launch And Resume

### 6.1 Role Policy

Mission card role policy remains the authority:

- `topology-supervisor`: entry role, owner-facing, no business code writes
- `hq`: coordinator, no business code writes
- `repair`: executor, `allowed_paths_only`
- `runner`: verifier, read-only
- `oracle`: reviewer, read-only
- `librarian`: evidence curator, read-only
- `scott`: scout/researcher, read-only

Launch metadata must include:

- `mission_id`
- `role`
- `session_id`, when known
- `script_path`
- `provider`
- `model`
- `thinking`
- `tools`
- `write_policy`
- `allowed_paths`
- `forbidden_actions`
- `permission_source`: path to Mission card or registry snapshot

For read-only roles (`runner`, `oracle`, `librarian`, `scott`), launch metadata must normalize to `write_policy: "deny_all_writes"` or an equivalent explicit read-only value, and `allowed_paths: []`. The global Mission `allowed_paths` may still exist, but it must not be copied into read-only launch metadata as writable scope.

Before writing or launching a role script, runtime must validate that launch metadata matches Mission policy. A mismatch must:

- block launch
- append an incident
- append a `launch_blocked` runtime event
- leave existing evidence untouched

Policy changes during a Mission are not retroactive. Already launched roles continue under the permission snapshot recorded at launch unless the owner parks/closes and relaunches them. Planned or not-yet-launched roles must validate against the latest Mission policy before launch.

### 6.2 Launch Modes

Supported launch modes:

- `print`: write script and return command.
- `direct_script`: operator executes generated script in the current terminal.
- `launch`: issue terminal app open command.

Acceptance meaning:

- `print` proves script generation only.
- `direct_script` can prove role startup if durable `alive_confirmed`, `session_alive`, registry, packet, or terminal log evidence exists.
- `launch` proves command issuance only unless the role later self-confirms.

Ghostty GUI launch is `evidence_degraded` on this Mac. It is not evidence-disabled, because commands may partially execute, but it is not stable enough to serve as role-alive evidence by itself.

### 6.3 Role Lifecycle Actions

Role lifecycle actions are separate from launch modes:

- `park`: mark a role intentionally paused; do not delete scripts, packets, or session records.
- `unpark`: return a parked role to `resumable` and require liveness confirmation before dispatch.
- `close`: mark a role closed for this Mission; do not send new work to that role.
- `replace`: close or park the existing role evidence, then create a new launch record.

Each action must append a session transition event and update the Mission registry role summary.

### 6.4 Resume Order

When resuming a Mission, Supervisor classifies every expected role:

1. Check fresh peer registry heartbeat. If fresh, mark `live`.
2. Check latest `alive_confirmed` session record. If within resume freshness window and not closed/failed, mark `resumable`.
3. Check script records. If script exists and Mission is not closed/archived, mark `resumable` with `needs_liveness_confirmation`.
4. If owner parked the role, mark `parked`.
5. If Mission is delivered/archived or role has closed record, mark `closed`.
6. Otherwise mark `stale`.

Supervisor must not send work to `resumable` or `stale` roles until liveness is confirmed or the role is relaunched.

### 6.5 Role-Specific Launch Rules

Default launch set:

- Mission creation starts only Supervisor.
- After owner approval, default first launch is HQ.
- Runner is launched when there is a concrete verification contract.
- Scott is launched when Pi/API/package research is explicitly needed.
- Oracle is launched after runner/scott evidence exists or when independent review is requested.
- Librarian is launched when evidence indexing or closeout artifact work is needed.
- Repair is launched only after a scoped fix need is identified and write boundaries are clear.

## 7. Inbox Cleanup And Stale Packets

Cleanup is evidence-preserving.

Allowed cleanup actions:

- mark packet lifecycle state
- write `packet-ledger.jsonl`
- write derived summary/index records
- filter active reads by Mission id
- ignore duplicate terminal/control packets
- hide stale packets from default dashboard counts

Disallowed cleanup actions without explicit owner gate:

- delete raw outbox/inbox files
- rewrite raw packet history
- delete artifacts
- delete ledgers
- delete Mission folders

Default active reads must include only packets where:

- `packet.mission_id === active_mission_id`
- packet state is not `closed`, `report_acknowledged`, `ignored`, `stale`, or `duplicate`
- packet type is actionable for the requesting role

`include_history=true` may show preserved stale or historical packets, but the output must label them as historical.

## 8. Evidence Path Convention

Every reviewable evidence item must have a mission-scoped path.

Path forms:

- `mission:<mission_id>/mission-card.json`
- `mission:<mission_id>/status-board.json`
- `mission:<mission_id>/runtime-events.jsonl#<event_id>`
- `mission:<mission_id>/sessions.jsonl#<record_id>`
- `mission:<mission_id>/packet-ledger.jsonl#<packet_id>`
- `mission:<mission_id>/<relative_artifact_path>`
- absolute filesystem paths are allowed in records, but human-facing summaries should prefer mission-scoped paths

Line numbers are optional because JSONL files are append-only and record ids are more stable.

## 9. Delivery And Rollback

### 9.1 Delivery

A Mission is `delivered` only when both exist:

- `closeout.md` or equivalent closeout artifact in the Mission folder
- runtime event `owner_acknowledged_delivery` or explicit owner acceptance record

The closeout must include:

- Mission id
- objective
- final verdict
- changed files or delivered artifacts
- tests/evidence paths
- remaining risks
- parked/closed role status

### 9.2 Rollback

Rollback is a decision-gated recovery path.

When a slice or Mission action fails:

1. Mark active slice or Mission `rollback_pending` or `blocked`.
2. Write incident with transport/business/inference evidence.
3. Preserve all raw evidence.
4. Ask owner/reviewer whether to revert code, add repair slice, park Mission, or abandon Mission.

No runtime tool should silently run destructive git or filesystem cleanup as rollback.

## 10. UI, Footer, And Status

`/topology` output must be current-Mission-first.

Required fields:

- active Mission id/title
- lifecycle state
- owner gate
- next action
- role counts by `live`, `resumable`, `stale`, `parked`, `closed`
- pending active packet count
- incident count
- closeout/artifact pointer when present

`/topology status` must show detailed paths.

Footer/widget support is `supported` for status/widget rendering based on the existing API audit, but exact multi-Mission UI composition is implementation detail. Native chooser UI via Pi dialog/custom UI primitives is `supported`; this round may keep a text-first picker for compatibility, with a future path to native chooser once the picker UX becomes the bottleneck.

## 12. Migration

### 12.1 Legacy Detection

If root `.pi/topology/mission-card.json` exists and `mission-registry.json` does not:

1. Read and validate root Mission.
2. Create `.pi/topology/missions/<mission_id>/`.
3. Copy root active files into that folder.
4. Write `mission-registry.json`.
5. Write `active-mission.json`.
6. Keep root files as active mirrors.
7. Append `mission_migrated` event.

If copied files are missing, create empty compatible files and record inference.

Migration-created empty JSON files must carry `_meta.inferred_empty: true`. Migration-created empty JSONL files must receive a first metadata row with `event_type: "migration_inferred_empty"` or equivalent. Reviewers must be able to distinguish "truly no events yet" from "legacy file was missing during migration."

### 12.2 Mirror Updates

Whenever active Mission canonical files change, update root compatibility mirrors for:

- mission card
- status board
- runtime event append
- incident append
- sessions append
- launch scripts

Implementation may centralize writes through path helpers to avoid double-write bugs.

## 13. Implementation Slices

Pi Coder implementation must use these slices unless reviewer changes the order:

1. Mission registry and per-Mission directory layout.
2. Supervisor mission picker and mission resume/create flow.
3. Session registry semantics for role session IDs and stale/alive evidence.
4. Inbox cleanup and stale packet marking.
5. Dashboard/status output for multi-Mission state.
6. Migration from current single-Mission layout.
7. Final dogfood Mission with direct generated-script launches.

Each slice handoff must include:

- `<slice_id>-notes.md` or equivalent implementation note
- changed files
- focused tests
- `npm run smoke` output from `packages/pi-topology`
- evidence paths
- local commit hash
- known risks or reviewer questions

Commit convention:

- one atomic commit per approved slice
- message format: `slice(<slice_id>): <summary>`
- no push unless owner explicitly asks

Slice dependencies:

| Slice | Depends on | Notes |
| --- | --- | --- |
| 1. Mission registry/layout | none | Foundation for all later slices. |
| 2. Supervisor picker/resume/create | 1 | Needs registry and active pointer. |
| 3. Session registry semantics | 1 | Can proceed after per-Mission paths exist. |
| 4. Inbox cleanup/stale packet marking | 1, 3 | Needs Mission identity and session/stale semantics. |
| 5. Dashboard/status output | 1, 3, 4 | Needs Mission, session, and packet classifications. |
| 6. Migration | 1, 2, 3, 4, 5 | Run after target paths and dashboard behavior are stable. |
| 7. Final dogfood | 1, 2, 3, 4, 5, 6 | Acceptance run after all migration and dashboard work. |

## 14. Test Requirements

Minimum focused tests:

- create registry from no-Mission workspace
- migrate legacy root Mission into per-Mission folder
- active pointer switches Mission without deleting old evidence
- `/topology` dashboard counts only active Mission pending packets
- stale session record older than freshness window is not live
- launch metadata permission mismatch blocks role launch
- cleanup marks stale/duplicate packets without deleting raw packet files
- direct generated-script lane still records the current verified runtime profile
- `npm run smoke` passes

Dogfood evidence:

- use `mode=print` to generate scripts
- verify generated role scripts contain the current verified runtime profile. As of 2026-06-17 this profile is `--provider minimax-cn --model MiniMax-M3 --thinking low`; future changes should update one baseline constant or record, not scatter string checks.
- execute generated role script directly
- require durable role alive evidence before marking role live
- do not use unattended Ghostty GUI launch as acceptance evidence

## 15. API Audit Follow-Up Items

Official Pi documentation already answers several API questions. The Step 3 API audit should record these answers and then verify local package behavior where needed.

1. Pi supports session switching by session file path via `ctx.switchSession(sessionPath, options?)`, not arbitrary role session id. The audit must decide whether Mission session records should store session file paths in addition to role session ids.
2. Pi supports `ctx.newSession(options?)` for in-extension session creation. The audit must decide whether this can be used without losing visible peer mesh semantics; current implementation remains generated visible scripts.
3. `pi.sendMessage` and `pi.sendUserMessage` inject into the current session. The audit should record that they are not cross-session role routing primitives.
4. `deliverAs` controls current-session delivery timing (`steer`, `followUp`, `nextTurn`). It does not replace local role-to-role packet routing.
5. Native chooser/custom UI primitives are supported. The audit should define when `/topology` should move from text-first picker to native chooser.
6. Footer/widget/status primitives are supported. Multi-Mission aggregation is implementation work, not an API gap.
7. Package skill/resource discovery is supported through package conventions and `resources_discover`; the audit should re-verify both installed package and `pi -e ./index.ts` development paths.
8. `session_shutdown` can observe Pi session lifecycle, but OMP business states such as `parked` and `closed` remain local protocol states.

None of these items should block the local file-driven implementation. They determine how much Pi-native behavior can replace or complement local protocol in later slices.
