## 3. 为什么不能直接从 Cave 深路径 `-e`

尝试：

```bash
omp models ls -e /Users/yuantian/Documents/Cave/Agent-Lab/Agent工具/OMP/pi-vs-claude-code移植/ports/coms-omp/extensions/coms-omp.ts
```

结果：OMP 能启动并列出 models，但 extension load 失败：

```text
Failed to load extension ... coms-omp.ts:
Cannot find module '@oh-my-pi/pi-tui'
```

原因：

- `coms-omp.ts` 静态 import 了 `@oh-my-pi/pi-coding-agent` 和 `@oh-my-pi/pi-tui`。
- 本机 OMP 运行依赖在 `~/.omp/node_modules/@oh-my-pi/`。
- Bun 从 Cave 深路径向上找 `node_modules`，不会自然走到 `~/.omp/node_modules`。
- `NODE_PATH=/Users/yuantian/.omp/node_modules` 对这个 Bun/OMP bundled runtime 不生效。

这不是 coms 协议问题，是 extension package resolution 问题。

## 4. 推荐嵌入方式

### A. 第一轮 smoke 推荐：放到 OMP home 的实验目录，默认加载 `lite` 入口

目标路径：

```text

```

这个路径的好处：

- 不会像 `~/.omp/agent/extensions/` 那样自动全局加载。
- 可以用 `-e ~/.omp/agent/experiments/coms-omp` 手动加载。
- 是 OMP 运行目录内部，后续可平滑升格为常驻 extension 或正式 plugin。

实测更新：

- 把完整端口放到 `~/.omp/agent/experiments/coms-omp` 后，仍然无法解析 `@oh-my-pi/pi-tui`。
- 这说明 OMP 的 legacy extension loader 不会把 `~/.omp/node_modules` 作为外部 extension 的可靠解析上下文；仅靠“搬近 OMP home”不够。
- 第一轮已新增 `extensions/coms-omp-lite.ts`，去掉 `@oh-my-pi/pi-tui` / `DynamicBorder` / rich renderer，只保留通信核心：
  - `--cname` / `--purpose` / `--project` / `--color` / `--explicit`

  - `coms_list` / `coms_send` / `coms_get` / `coms_await`
  - `session_start` / `agent_end` / `session_shutdown`

### A2. 完整 UI 端口的后续路线

完整 `coms-omp.ts` 仍保留在 Cave 基线里，但它现在不适合作为第一轮 smoke 入口：

1. 它静态依赖 `@oh-my-pi/pi-tui`，当前本机 OMP 对外部 extension 的解析不稳定。
2. 它包含 live pool widget / rich renderers，这些是体验层，不是 pi-to-pi 原语的必要条件。
3. OMP 更新频率高，TUI renderer 和 ExtensionAPI 漂移风险比 socket/registry 工具层高。

后续若要恢复完整 UI，有两条稳路：

- 把 UI 端口做成 OMP 内置/插件级 package，让依赖跟 OMP runtime 一起解析。
- 或者把 UI renderer 改成只使用 OMP 当前稳定公开 API，不直接 import `pi-tui`。

## 4.5 装 Pi、装 OMP，还是改 OMP 本体

当前判断：**不要把主线放在 Pi；第一轮装进 OMP 运行目录；中期考虑做 OMP-native package，不急着改 OMP 本体。**

原因：

- Pi 原生跑这套更容易，但会失去 OMP 的 coding 增强面：AST / LSP / OMP task / Agent Hub / 当前本机模型配置。我们的目标是 ekunAi smoke 和后续真实 coding 工作流，所以主战场应是 OMP。
- 直接改 OMP 本体可以最稳地解决依赖解析和 UI 接入，但会把维护成本放大到每次 OMP 升级的源码级 rebase。现在还没证明通信原语在 ekunAi 工作流里的收益，不宜先改 core。
- 放进 OMP 运行目录的 extension/package 是最合适的中间层：能真实使用本机 OMP，又能用 `-e` 控制实验范围；验证成功后再升格为 `~/.omp/agent/extensions/coms-omp` 或正式 plugin。

分层路线：

| 阶段 | 目标 | 入口 |
|---|---|---|
| Phase 1 | 跑通本机平级通信 smoke | `~/.omp/agent/experiments/coms-omp` + `coms-omp-lite.ts` |
| Phase 2 | ekunAi 两终端实战 smoke | `planner` / `implementer` 同项目 `ekunAi-smoke` |
| Phase 3 | OMP-native package | 固化 manifest、测试、安装脚本、兼容 smoke |
| Phase 4 | 评估是否改 OMP core | 只在 extension loader / package resolution 成为长期阻塞时考虑 |

### B. 项目级常驻：在 ekunAi `.omp/config.yml` 写 extensions

只在 smoke 项目验证稳定后再做。示意：

```yaml
extensions:
  - ~/.omp/agent/experiments/coms-omp
```

注意：`ekunAi` 是 Cave 外项目。本 Cave agent 默认不能直接写它；需要用户明确授权或由用户自己改。

### C. 全局自动加载：放到 `~/.omp/agent/extensions/`

不推荐第一轮使用。

```text
~/.omp/agent/extensions/coms-omp/
```

优点：不用每次 `-e`。  
缺点：所有 OMP session 都会加载 `coms`，容易污染无关项目和 registry。

### 6.6 ekunAi HS smoke 后的拓扑修正：Hub-and-Mesh

2026-06-14 晚间，`ekunAi` 的 HS Code fresh Direct smoke 用四个 OMP session 进行实战演练后，对第 6 节的“planner 星型分发”形成修正：星型调度可控，但过于接近传统中心化 subagent 队列，没有充分利用 `coms-omp-lite` 的长期在线 peer 通信能力。

新的推荐形态不是纯中心化，也不是完全去中心化，而是 **Hub-and-Mesh**：

- `hq-control` 是唯一 goal owner 和权限 gate，负责目标、边界、止损、授权、最终 verdict。
- `smoke-runner` 负责 runtime execution 和 artifact evidence。
- `oracle-reviewer` 负责高智力审查、验收红线、证据索取和 P0/P1 风险判断。
- `repair-engineer` 负责 root-cause triage，以及在 HQ 授权后的 scoped fix。

核心变化：HQ 不再做所有消息的“邮差”。非 HQ session 可以直接互相 `coms_send`，但这些横向边只用于信息流，不用于授权流。

```text
Authority edges:
HQ -> Smoke / Oracle / Repair
- start / stop
- enter next case
- authorize file or DB changes
- approve cache invalidation or fresh budget
- final PASS / ACCEPTABLE_NEEDS_REVIEW / HARD_FAIL

Information edges:
Oracle <-> Smoke
- evidence request
- review checklist
- missing proof

Repair <-> Smoke
- artifact request
- reproduction detail
- classification audit / declaration row detail

Repair <-> Oracle
- repair acceptance boundary
- risk review
- test gap review
```

因此正式工作流纪律应修正为：

```text
本工作流不是单向派单。除 hq-control 的最终裁决权外，smoke-runner、oracle-reviewer、repair-engineer 可以互相直接 coms_send 提问、索证、复核和补充判断。

但以下动作必须等待 hq-control：
1. 停止当前 case / 停止整批。
2. 进入下一 case。
3. 修改任何文件或 DB。
4. 清 cache、重启服务、消耗额外 fresh 预算。
5. 将结果记为最终 PASS / ACCEPTABLE_NEEDS_REVIEW / HARD_FAIL。

横向通信是 information edge，不是 authority edge。
```

### 6.7 Persistent role mesh + ephemeral task swarm

这轮讨论进一步沉淀出一个中期推荐架构：**Persistent role mesh + ephemeral task swarm**。

含义：

- Persistent role mesh：长期存在的 OMP sessions 承担稳定角色和责任边界，例如 HQ、Smoke、Oracle、Repair。它们保留项目状态、角色人格、协作协议和长期目标。
- Ephemeral task swarm：每个 persistent session 内部仍可使用 OMP/Codex 的 task/subagent 能力，派生短生命周期的 reader、prober、tester、diff reviewer，只处理局部问题，结束后把结果汇总回所属 persistent session。

推荐结构：

```text
Persistent role mesh
- hq-control: goal/state/authority
- smoke-runner: runtime execution and evidence
- oracle-reviewer: audit and acceptance criteria
- repair-engineer: root cause and scoped implementation

Ephemeral task swarm
- each persistent session may spawn short-lived local readers/probers/testers
- no ephemeral task owns final project state
- final authority remains with hq-control
```

这个结构的价值不是“更多 agent 更聪明”，而是上下文分治和认知模式分治：

- HQ 保持项目级 working memory，不被代码细节污染。
- Smoke 保持运行态和 artifact 细节。
- Oracle 保持审计人格，不被修复冲动污染。
- Repair 保持 debug/implementation 上下文，但不拥有授权权。
- 短生命周期 subagents 负责局部搜索，避免长期 session 堆积无关细节。

与 `ekunAi` 税则知识库的混合检索类似：纯向量、纯图谱、纯数据库都不够；agentic coding 也不应纯单 session、纯中心化或纯网状。当前最可用的生产形态是：

```text
mesh communication + centralized authority + single-writer implementation + independent review
```

### 6.8 ekunAi HS fresh smoke 实战观察

本轮四 session 在 `ekunAi` HS Code fresh Direct smoke 中抓到一个真实 P0：cached/pre-smoke replay 已经显示 repo seed 能把 `MOUNT ADAPTER` 从错误 `8529` 收回 `8517`，但 fresh production runtime 仍命中 runtime `knowledge/kb.db` 中 active 的 `mountadapter -> 8529909090` 污染 alias。

这个失败验证了 mesh 的价值：

- `smoke-runner` 产出 fresh job evidence：`route=direct_llm_items`、`cache_hit=false`、job/artifact path。
- `oracle-reviewer` 抓住红线：`MOUNT ADAPTER` 回退 `8529`，第一批立即 stop-loss。
- `repair-engineer` 将 root cause 收敛到 seed/runtime DB drift，而不是 profiler/API/cache/输入质量。
- `hq-control` 保留权限 gate：停止扩批，不进入 `test#14` / `test#25-1`，先组织受控 DB seed alignment。

后续对 `knowledge/kb.db` 执行 table-driven seed replay 和 exact cache invalidation 后，`test#10` fresh rerun 关闭 P0：新 job `2783583f3051`，`cache_hit=false`，关键行不再回退到 `8529` family。剩余 `needs_review` 不等于 full passed，仍需如实记录。

这说明 Hub-and-Mesh 最适合的任务不是简单单文件改动，而是 runtime-heavy、review-heavy、需要止损和证据审计的生产级 agentic coding 工作。


### 6.9 Mesh 协作可靠性规则：late result 与自主权限包

`ekunAi` HS smoke 实战还暴露了一个调度层细节：`coms_await` 超时不等于 peer 没有结果。一次 micro-triage 中，HQ 先报告 `repair-engineer` 超时未返回，但随后又拿到了 `repair-engineer last message`，且该结果质量很高并与最终判断一致。

因此正式规则新增：

```text
如果 peer await 超时，HQ 必须主动 coms_get 或重新向该 peer 索取 last result；不能仅凭 await timeout 宣称“无结果”。

推荐处理顺序：
1. 标记该 peer 为 late/pending，而不是 failed。
2. 调用 coms_get 或向 peer 发送“请返回你的 last result / current status”。
3. 如果 late result 到达且包含有效证据，必须并入最终 judgment。
4. 如果 late result 与已形成结论冲突，HQ 应组织一次短对账，而不是静默忽略。
5. 只有在二次索取仍无响应时，才把该 peer 标为 timeout，并说明该结论缺少哪个角色的意见。
```

这条规则的意义：在 persistent role mesh 中，每个 session 都是长期角色，不是一次性 subagent。长期角色可能因为工具调用、shell、外部 API 或人工查看 artifact 而延迟，HQ 的职责是维护结果完整性，而不是把 transport timeout 误判为认知失败。

同一轮还形成了更高自主度的授权包。后续生产级 agentic coding 可按以下分层授权：

```text
Autonomous execution allowed:
- run focused smoke / probes / read-only SQL
- restart local API/frontend runtime when owner has declared no active users
- exact cache invalidation for the current case only
- table-driven seed replay with backup and before/after evidence
- focused tests, harness consistency, artifact inspection

HQ approval still required:
- enter next batch beyond the current owner-approved scope
- modify files or DB outside the granted autonomy envelope
- clear global caches
- change runtime DB without backup
- commit / stage changes
- declare final closeout

Owner approval still required:
- destructive file deletion
- production-facing promotion
- broad cache wipe
- git commit/push
- relaxing business gates such as LOW adoption
```

建议把这个称为：**bounded autonomy envelope**。它与 Hub-and-Mesh 的关系是：

```text
mesh communication gives speed
bounded autonomy gives progress
centralized authority keeps risk bounded
single-writer implementation keeps git sane
independent review keeps conclusions honest
```

### 6.10 当前推荐的生产级 OMP agentic coding 协议

综合 6.6-6.9，当前中期推荐协议如下：

1. HQ 维护 goal，不维护所有底层代码细节。
2. Persistent role mesh 保持四类长期角色：HQ / Smoke / Oracle / Repair。
3. Peer-to-peer 通信默认允许，但只作为 information edge。
4. Authority edge 仍只从 HQ 发出，owner 可进一步限制或扩大 HQ 的 autonomy envelope。
5. 所有 runtime/cache/DB 动作必须记录 exact target、before/after evidence 和 rollback plan。
6. 对 fresh smoke，`status=cached` 永远不能被包装成 fresh success。
7. 对 review，`needs_review` 不能被包装成 full passed。
8. 对 coding，保持 single-writer：同一时间只有一个角色被授权改同一层文件。
9. 对 late peer result，必须二次索取并并入判断。
10. 对 git，最终由 HQ 汇总 dirty worktree 分桶，但 stage/commit 需 owner 明确批准。

这套协议适用于 runtime-heavy、review-heavy、需要止损和证据审计的任务；对单文件小修仍不建议启用四 session mesh，避免协调成本超过收益。
