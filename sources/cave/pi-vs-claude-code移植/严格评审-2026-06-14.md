## 4. 已知 API 变动与维护判断

### 4.1 `--list-models` 已过时

历史 `coms-omp` 迁移日志记录过：

```bash
omp --list-models "" -e extensions/coms-omp.ts
```

但 OMP `15.12.4` changelog 明确：

- Removed top-level `--list-models`
- Migrated model listing to `omp models`

维护动作：

- 所有 smoke / 文档命令改为 `omp models ls -e extensions/coms-omp.ts`
- 旧 smoke 作为历史记录保留，但不能再当当前验证标准

### 4.2 extension-owned flags 问题已有上游修复迹象

历史 `coms-omp` 迁移日志记录：

- OMP `15.5.15` 下 `--name` / `--project` 等 extension flag 会被当成 prompt 文本

OMP `15.12.5` changelog 与源码显示：

- `parseArgs` 现在会记录 unknown flag
- `runRootCommand` 会在 extension load 后做 extension-aware reparse
- extension-registered flags 仍可通过 reparse 清除 unknown 状态
- 真正 unknown flags 会 fail fast，而不是吞成 prompt

源码证据：

- `packages/coding-agent/src/cli/extension-flags.ts`
- `packages/coding-agent/src/main.ts`
- `packages/coding-agent/src/extensibility/extensions/types.ts`
- `packages/coding-agent/src/extensibility/extensions/loader.ts`

维护判断：

- 这是好消息。`coms-omp` 的 `registerFlag/getFlag` 路线仍可继续。
- 但必须用 `v15.12.6` 重新跑 flag parse smoke，确认 `--name` / `--project` 真正进 `pi.getFlag()`。

## 5. 我们是否有能力维护这些 TS 代码

结论：有，但必须收窄维护面。

可维护的原因：

- `coms` 核心协议不是 OMP 私有能力，主要是 Node built-ins。
- OMP 当前仍保留公开 ExtensionAPI，并且向后兼容了 typebox shim。
- OMP 源码本身有 extensibility tests，API 变动通常能从 changelog / types.ts 观察到。
- 历史 `coms-omp` 已完成过真实 two-process transport smoke 与在线模型 E2E。

主要风险：

| 风险 | 等级 | 处理 |
|---|---|---|
| OMP 高频版本导致 CLI / flags / event 名变化 | 高 | 每次升级跑 extension smoke；订阅 `packages/coding-agent/CHANGELOG.md` |
| TUI 组件或 renderer 内部变化 | 中高 | P0 移植先保核心功能，UI widget 降级可用 |
| session storage / Agent Hub 内部 API 变化 | 高 | 不依赖内部路径，除非单独封装 adapter |
| tool schema 方式变化 | 中 | 优先 `pi.zod`，保留 `pi.typebox` shim only for legacy |
| 与 OMP 原生 `task` / `irc` / Agent Hub 重叠 | 中 | 不重复建设；用差异表决定是否移植 |

维护策略：

1. 为每个 port 建 `compat-smoke`：
   - extension 能 load
   - flags 能 parse
   - tools 能注册
   - 关键 tool 能在无模型 / fake context 下执行
   - 对 `coms` 额外跑 two-process ping/prompt smoke
2. 每次 OMP tag 更新只看三处：
   - `packages/coding-agent/CHANGELOG.md`
   - `packages/coding-agent/src/extensibility/extensions/types.ts`
   - `packages/coding-agent/src/cli/args.ts` / `cli/extension-flags.ts`
3. UI 与核心协议拆开：transport/tools 先稳定，widget 后增强。
4. 使用 OMP home：`~/.omp/coms`，不要复用 `~/.pi/coms`。
